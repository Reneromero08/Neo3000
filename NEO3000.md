# Neo3000 Architecture

## Mission

Neo3000 is a local inference engine for Agents-A1. It begins from a known-good `llama.cpp` runtime, preserves OpenAI-compatible streaming for Pi, and modifies the compute path toward catalytic, phase-native, recursively reusable inference.

The practical target is not a demonstration that merely restores a file-backed tape. The target is to reduce the amount of fresh computation required for each useful token while preserving model capability and interactive stability.

## Priority order

```text
1. Correct model execution
2. Pi-compatible streaming
3. Daily-driver stability
4. Short-context speed
5. Long-context speed
6. Compute observability
7. Catalytic compute experiments
8. Bounded recursive self-improvement
```

No later priority may silently regress an earlier one.

## Initial machine and workload

The first deployment target is:

- Windows 11
- Ryzen 9 5900X
- 32 GB system RAM
- NVIDIA RTX 3060 12 GB
- Agents-A1 GGUF
- context window up to 65,536 tokens
- Pi as the interactive agent harness

The current comparison runtime is LM Studio, observed around 8 tokens per second at shorter context and around 3 tokens per second near 40,000 tokens. These numbers are provisional until reproduced with a fixed prompt and configuration.

## Catalytic compute

Neo3000 treats an algorithm as a local trace through a larger process-object. The runtime should progressively replace disposable serial work with reusable relational state.

```text
borrow existing compute state
-> transform through one or more trajectories
-> extract a surviving result or invariant
-> restore or close temporary state
-> retain only lawful durable state
```

The main compute programs are:

### 1. Single-model catalytic speculation

Use reduced-depth or alternate-path traversals of Agents-A1 itself to draft several future tokens, then verify them together with the full model. Rejected futures disappear with their branch state. Accepted tokens amortize one expensive target traversal across multiple emitted tokens.

### 2. Holographic long-context state

Keep recent context exact while transforming distant context into an executable relational representation. The current query should illuminate and reconstruct relevant interactions without scanning every old token as an equal explicit position.

### 3. Layer-orbit closure

Observe whether hidden-state relations stabilize before the final layer. When a predeclared relational closure condition is reached, skip or cheaply verify the remaining layers rather than executing them blindly.

### 4. Reusable dirty compute arenas

Move temporary inference buffers toward borrow, transform, extract, and restore semantics. Begin with exact reversible state transitions and high-traffic scratch operations rather than claiming the full transformer is reversible.

### 5. Recursive compute amplification

Allow accepted computational structures to return as operators or state for later inference. Measure the ratio between equivalent baseline compute and fresh compute actually executed.

## Minimal runtime shape

The imported source remains close to upstream until baseline parity is proven. Native Neo3000 code should later concentrate under a small surface:

```text
neo/
  state/
  speculation/
  phase/
  topology/
  kernels/
  trace/

tools/
  neo-bench/
  neo-loop/
```

These directories should be created only when the first real implementation requires them.

## Experimental law

Every performance intervention must identify:

1. the exact expensive operation
2. the state being borrowed or reused
3. the transformation performed
4. the output or invariant extracted
5. the restoration or closure law
6. the expected speed mechanism
7. the quality and stability gates
8. the measurement that would reject the intervention

Do not collapse phase-native, topological, spectral, or relational objects into scalar candidate ranking unless that reduction is explicitly under test.

## Checkpoint sequence

### Checkpoint 0: Baseline parity

Import the pinned upstream source, build CUDA release binaries, serve Agents-A1 directly to Pi, and reproduce fixed performance measurements.

### Checkpoint 1: Compute map

Instrument prompt processing, token generation, layer timing, attention, MoE routing, CPU and GPU transfers, KV traffic, and memory pressure with tracing that can be compiled out.

### Checkpoint 2: Catalytic self-speculation

Implement one-model reduced-compute drafting and batched target verification. The success metric is accepted future tokens per expensive target traversal.

### Checkpoint 3: Long-context slope

Measure and reduce the decline in decode speed from short to long context. Start with exact recent state plus an experimental distant-context side channel.

### Checkpoint 4: Layer-orbit closure

Test whether relational state becomes predictively closed before the final layer and whether remaining work can be removed without material quality loss.

### Checkpoint 5: Autonomous candidate loop

Agents-A1 edits a candidate worktree, builds it, runs a fixed benchmark, records the result, and continues under bounded goals while the stable server remains available.

### Checkpoint 6: Native catalytic kernels

Replace selected hot-path state transitions with exact borrow, transform, extract, and restore operations.

## Primary metrics

- prompt processing tokens per second
- decode tokens per second
- time to first token
- rolling minimum decode speed
- context degradation ratio
- accepted speculative tokens per target pass
- fresh FLOPs or operator calls per emitted token
- CPU to GPU transfer volume
- RAM and VRAM peaks
- output correctness
- Pi tool-call validity
- crash and recovery behavior

The central long-context metric is:

```text
context degradation ratio = long-context decode TPS / short-context decode TPS
```

The central catalytic metric is:

```text
compute amplification = equivalent baseline compute / fresh compute executed
```

## Non-goals for the foundation phase

- MCP integration
- AGS governance import
- a web dashboard
- a plugin ecosystem
- a semantic database
- distributed serving
- automatic promotion to stable
- broad documentation generation
- speculative claims without a runnable model

Neo3000 earns complexity only when it removes more computation or operational friction than it adds.
